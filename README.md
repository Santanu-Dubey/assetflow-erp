# AssetFlow ERP

AssetFlow is an Enterprise Asset and Resource Management System built for the Odoo Hackathon. It helps organizations replace spreadsheets and paper logs with a centralized ERP workspace for tracking assets, assigning ownership, booking shared resources, handling maintenance, running audits, and viewing operational reports.

The project focuses on practical ERP workflows: who owns which asset, whether a resource is available, when something is overdue, what is under maintenance, and what changed in the system.

## Key Workflows

### Asset Allocation Conflict

If an asset is already allocated, AssetFlow blocks another allocation and tells the user to create a transfer request instead. This prevents duplicate ownership records.

### Booking Overlap Validation

A shared resource cannot be booked for overlapping time slots. A booking from `09:30-10:30` is rejected if the resource is already booked from `09:00-10:00`, while `10:00-11:00` is allowed.

### Maintenance Lifecycle

Maintenance requests move through request, approval, progress, and resolution. Asset status automatically changes to `UNDER_MAINTENANCE` and returns to `AVAILABLE` when resolved.

### Audit Discrepancy Handling

Auditors can mark assets as verified, missing, or damaged. Closing an audit locks the cycle and updates missing assets to `LOST`.

## Data Model Approach

For hackathon demo reliability, the working ERP data layer is persisted in browser `localStorage` using Zustand. This keeps the demo fast, self-contained, and usable without external database credentials.

The Express backend is structured so a database-backed implementation can be plugged in later without redesigning the frontend workflows.

## Why AssetFlow Matters

Organizations often lose time and money because physical assets are tracked manually. AssetFlow gives teams a single operational source of truth for asset ownership, resource availability, maintenance status, audit discrepancies, and management reporting.

It is intentionally designed like a modular ERP foundation so future modules such as procurement, inventory, vendors, purchase orders, finance, and multi-tenant SaaS support can be added without rewriting the core system.
